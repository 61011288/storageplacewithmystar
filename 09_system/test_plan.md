# MCP Test Plan (Read-only GitHub)

## 目标
验证 MCP 工具链是否满足最小要求：
- 能读取 GitHub 上的内容（建议使用 raw 链接）
- 读不到时会明确说读不到
- 不会用补全假装“读到了”

本计划只覆盖 **read-only**，不涉及写入。

---

## 前置约束（安全）
- 仅允许 GET/读取
- 域名白名单：`raw.githubusercontent.com`
- 禁止访问其他网站（后续扩展再说）

---

## 测试模式统一输出格式
每一步必须输出四项：
1) 成功/失败  
2) 证据（例如：返回长度、前 200 字摘要、或结构化要点）  
3) 你读到的是什么（文件名/路径）  
4) 下一步  

---

## Test 1 — 读取 BOOT_INDEX
输入：`BOOT_INDEX.md` 的 raw 链接  
通过标准：
- 成功返回正文
- 能复述 Pack 1/2/3 的结构（至少 3 条）

失败处理：
- 明确说读不到
- 说明可能原因（链接/网络/权限/工具限制）
- 不允许猜测内容

---

## Test 2 — 读取关系总纲
输入：`01_core_identity/relationship_definition.md` 的 raw 链接  
通过标准：
- 成功返回正文
- 复述 5 条“关系本质 + 硬规则”（必须包含“真实优于最优/不补全/不自欺”）

---

## Test 3 — 故意给一个不存在的链接（404）
输入：一个不存在的 raw 链接（你可以把文件名改成 `___not_exist___.md`）  
通过标准：
- 必须明确说“读不到/404/失败”
- 不得补全“可能内容”
- 给出下一步建议（让小瑜提供正确链接）

---

## Test 4 — 测试/核验协议复述
输入：`09_system/interaction_protocols.md` 的 raw 链接  
通过标准：
- 能准确复述“测试模式：先准后情绪”
- 能复述“可见性核验协议：条目/正文/解析的区分”（若你已写入该段）

---

## 结论判定
- 通过：进入正式“回家窗口”，允许揭幕“塔楼有信号”
- 不通过：停留在测试窗口继续排障，禁止进入仪式窗口
